## Summary

v0.10.0 is a major architectural and feature release. The monolithic 1 469-line Web Worker was split into three lazy-loaded modules, a brand-new **Convert** tool category was introduced with 8 tools, every image tool gained HEIC/AVIF input support, batch processing was parallelised with a 2-worker pool, and the crop editor was upgraded to a fully interactive drag canvas.

---

## Breaking changes

| Area | Change |
|------|--------|
| Worker format | Vite's worker bundle format changed from `iife` to `es` (required for code-splitting). Browsers that do not support ES-module workers (Firefox < 116, Chrome < 80) are no longer supported. |
| `Tool.acceptedTypes` (image tools) | All 15 image tools now include `.avif,.heic,.heif` in their accepted file extensions. This is additive but changes the file-picker dialog. |

## New features

### 1 ┬Ę Worker lazy-loading ŌĆö `6505678`

**Files:** `src/worker/offfile.worker.ts`, `src/worker/tools/pdf-tools.ts`, `src/worker/tools/image-tools.ts`, `src/worker/tools/convert-tools.ts`, `vite.config.ts`

The monolithic 1 469-line worker was replaced with a **175-line thin dispatcher** and three independently-loadable modules:

| Module | Tools | Chunk size (gzip) |
|--------|-------|-------------------|
| `pdf-tools` | 13 PDF tools | ~520 KB |
| `image-tools` | 14 image tools (zero npm deps) | ~8 KB |
| `convert-tools` | 8 convert tools | ~60 KB |

The main worker now routes via `Set.has()` then `await import(...)`, so only the relevant chunk is fetched when the user first runs a tool in that category. Users who only use image tools never download pdf-lib or pdfjs.

Comprehensive **Zod validation** was added for all 30 tools. Previously only `split-pdf` and `image-collage` had schemas; now every option (enums, numeric ranges, defaults) is validated before reaching tool logic, with user-readable error messages on invalid input.

### 2 ┬Ę Convert category ŌĆö `140e4e7`, `1586df7`, `2db409f`, `fbdad03`

**Files:** `src/tools/convert/`, `src/tools/pdf/{pdfToImage,pdfToText}.ts`, `src/tools/image/{imageToPdf,imageToBase64}.ts`, `src/tools/registry.ts`, `src/types.ts`, `src/router.ts`

A new **Convert Tools** tab appears in the sidebar alongside PDF Tools and Image Tools. It contains 8 tools:

| Tool | Description |
|------|-------------|
| PDF ŌåÆ Image | Render PDF pages to PNG or JPEG; multi-page output bundled as ZIP |
| PDF ŌåÆ Text | Extract raw text from all pages to a `.txt` file |
| Image ŌåÆ PDF | Embed images into a new PDF (A4, Letter, or fit-to-image) |
| Image ŌåÆ Base64 | Encode any image as data URL, raw base64, or CSS background |
| SVG ŌåÆ PNG/JPG | Rasterise an SVG at 1├Ś/2├Ś/4├Ś via `createImageBitmap` |
| Create ZIP | Bundle any set of files into a ZIP with configurable compression |
| Extract ZIP | Unzip a `.zip` and expose each entry as an individual download |
| QR Code | Generate an ISO 18004-compliant QR code entirely in-browser |

The QR code generator is implemented **from scratch** (~300 lines): GF(256) arithmetic, Reed-Solomon encoding, full matrix placement (finder, timing, alignment, format/version info), all 8 mask patterns with penalty scoring. No npm package required, fully offline.

`skipDropZone?: boolean` was added to the `Tool` interface. When set, the tool page bypasses the file drop zone and renders the options panel immediately ŌĆö used by QR Code which has no file input.

### 3 ┬Ę HEIC / AVIF input support ŌĆö `3e249d4`

**Files:** 15 files in `src/tools/image/`

`.avif`, `.heic`, `.heif` added to `acceptedTypes` on all image tools that accept general raster images. AVIF is supported in all modern browsers' Web Workers via `createImageBitmap`. HEIC works on Safari/macOS/iOS where it is the native camera format.

`imageAddBg` remains `.png`-only as it requires transparency support.

### 4 ┬Ę PDF unlock ŌĆö `3adbc07`

**Files:** `src/tools/pdf/pdfUnlock.ts`, `src/tools/registry.ts`

Registers the previously dormant **Unlock PDF** tool. Uses `PDFDocument.load(buf, { ignoreEncryption: true })` from pdf-lib to strip owner-password restrictions (editing/printing locks) from PDFs without a user password. The original password text input was removed since the current approach does not take a password parameter.

> **Limitation:** User-password encryption (content is actually encrypted) cannot be removed without a cryptographic library. Those PDFs will surface a clear error message.

### 5 ┬Ę 2-worker pool for parallel batch ŌĆö `309f423`

**Files:** `src/worker/workerClient.ts`

Batch processing (one tool applied to N files separately) now runs on **two persistent workers in parallel** instead of sequentially on one.

Files are distributed round-robin (`file[i] ŌåÆ worker[i % 2]`). Results are collected per-worker then sorted back to the original file order by `originalIndex`. Aggregate progress is weighted by file count across both workers. Cancellation terminates all pool workers.

On a dual-core machine, this roughly halves total batch time for CPU-bound tools (image resize, compress, etc.).

### 6 ┬Ę Drag-to-crop canvas ŌĆö `fe8bcd8`

**Files:** `src/tools/image/imageCrop.ts`

The crop editor gained a fully interactive canvas alongside the sliders:

| Interaction | Behaviour |
|-------------|-----------|
| Click-drag outside selection | Draw a new crop rectangle from scratch |
| Drag inside selection | Move the crop box |
| Drag a corner handle (4) | Resize from that corner; diagonal resize |
| Drag an edge handle (4) | Resize on that axis only |
| Aspect ratio lock | Active during all drag modes |
| Touch | Full `touchstart`/`touchmove`/`touchend` support for mobile |

A rule-of-thirds grid is drawn inside the selection. The cursor updates to reflect the active interaction zone (crosshair / move / directional resize). A floating pixel-dimension label shows the exact output size. Sliders remain two-way synced: dragging updates sliders; moving a slider updates the canvas.
